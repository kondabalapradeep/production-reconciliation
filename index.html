Method & rules — what every column means

Data sources

  • Sage Item Transaction History — the only quantity source. Production = JR (work ticket receipts) + BR (BOM production receipts) + IA (internal adjustments) netted against produced items — so an item produced 2,126,030 with a −2M adjustment shows its real net. Material consumption = JI (work ticket issues) + BI (backflush issues), plus IA on materials that are not produced items. Sales, transfers, and PO receipts are ignored.
  • Bill of Materials — BM_BillDetail gives the standard quantity of each component per unit produced (material lines only; labor/overhead lines excluded; scrap % intentionally NOT applied). CI_Item gives each item's description and LastTotalUnitCost.

Finished Goods Produced tab — column by column

  • Lot / Work Ticket(s) — the production run identifiers (Sage work ticket numbers) behind this item's JR receipts in the selected period. The extract has no separate lot-number field, so work tickets serve as the lot codes.
  • Sage Net (JR+BR+IA) — units produced per Sage, net of internal adjustments on the item.
  • Std Unit Cost (BOM) — the standard material cost to make ONE unit (BOM roll-up). Actual Unit Cost — the real material cost per unit this period: allocated actual cost ÷ units produced. Compare the two to see the per-unit impact.
  • Theoretical Cost — what those units should have cost in materials: units × BOM standard cost (the BOM roll-up: every component quantity × its unit cost, through every BOM level).
  • Actual Cost (Materials) — what the materials actually consumed for this item cost. Each component's real JI+BI consumption is allocated to the finished goods that required it (in proportion to their BOM demand), so if cans ran 13-per-case but sugar ran even higher, each shows up at its own real rate. Batch over-consumption flows through: a batch's real material cost per unit replaces its standard cost when valuing the finished goods that use it.
  • Cost Variance — Actual − Theoretical. Positive = materials cost more than BOM standard (scrap, waste, overfill); negative = less was consumed/posted than standard.
  • Var % — Cost Variance ÷ Theoretical Cost.
  • Ordering: Aerosol → Pops → Rainbow/Drinks → Water 1 Gal → Water 2.5 Gal → Batches always last. Categories inferred from item code prefix (A, P, H, W, L).

Raw Material Consumption tab — column by column

  • Unit Cost — the component's LastTotalUnitCost from CI_Item, used for all dollar columns on this tab.
  • Used In (Finished Goods) — the finished goods this component flowed into this period. Batches are skipped: components of a batch show the finished goods that consume the batch.
  • Theoretical (BOM) — Sage production × BOM quantity per unit. Example: 100 cases produced × 12 cans = 1,200 cans expected.
  • Theoretical Cost — Theoretical qty × unit cost (CI_Item LastTotalUnitCost).
  • JI+BI Issues / Adjustments (IA) / Total Actual — posted consumption, plus internal adjustments on the material shown in their own column (a negative on-hand adjustment = material that left stock without an issue entry, so it counts as consumption), and their sum. Deviation, costs, and status all use Total Actual. IA on items that are themselves produced (finished goods, batches) is netted on the production side instead, so nothing is counted twice.
  • Actual Cost — Actual qty × unit cost.
  • Deviation / Deviation Cost / Deviation % — Actual − Theoretical, in units, dollars, and percent. Because scrap is not in the theoretical, a healthy line runs slightly positive — that overage IS the real scrap/waste cost.
  • Status — IN LINE ≤5% · MODERATE 5–15% · OVER >15% · UNDER <−15% · NO BOM DEMAND (consumed but no produced item calls for it) · NOT CONSUMED (BOM demand but nothing posted).

Other rules

  • Day filter restricts both production and consumption to that transaction date. Issues and adjustments often post on a different day than the lot they refer to, so single days run noisier than the full period — the full-period view is the authoritative net; day views show what posted that day.
  • IA routing is fixed per item, not per day: an item that is ever produced (has JR/BR anywhere in the file) always has its adjustments netted against production — even when the adjustment posts on a different day than the lot. The extract carries no lot-number field on adjustments, so lot-level matching is not possible until the Sage lot distribution data is added to the export; every adjustment is still counted exactly once.
  • Consumption with no BOM demand cannot be attributed to any finished good, so it appears on the Raw Materials tab but not inside any item's Actual Cost.
  • Nothing uploads to any server — files are read locally in the browser. The shareable snapshot embeds the computed data so viewers need no files.
